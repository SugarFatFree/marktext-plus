## 2. 快捷键系统修复

### 2.1 问题分析

#### 焦点传播机制
Flutter 的键盘事件通过 `Focus` widget 树传播。事件从获得焦点的节点向上冒泡，直到某个节点返回 `KeyEventResult.handled`。

```
Widget 树层级：
  MaterialApp
    └─ HomeScreen
        └─ Focus (顶层，autofocus: true)  ← 最后一道防线
            ├─ MenuBar                      ← shortcut 属性仅在此子树内生效
            │   ├─ SubmenuButton (文件)
            │   ├─ SubmenuButton (编辑)
            │   │   ├─ MenuItemButton (查找, shortcut: Ctrl+F)  ← 无法触发
            │   │   └─ MenuItemButton (替换, shortcut: Ctrl+H)  ← 无法触发
            │   └─ ...
            └─ Editor
                └─ TextField (当前焦点)    ← 用户在此输入
```

#### 根因
`MenuBar` 的 `shortcut:` 属性注册的快捷键仅在 `MenuBar` 的 widget 子树内有效。当焦点在 `TextField` 中时，键盘事件不经过 `MenuBar`，因此快捷键无法触发。

#### source_editor.dart 排除
`source_editor.dart` 的 `_handleKeyEvent` 方法检查了 `_autoPairs` 映射（`(`、`[`、`{`、`"`、`'`、`` ` ``、`*`、`~`），对于 `f`、`h` 等非映射字符直接返回 `KeyEventResult.ignored`，不会错误消费事件。

### 2.2 解决方案

在 `home_screen.dart` 的顶层 `Focus` widget 的 `onKeyEvent` 中添加快捷键处理：

```dart
Focus(
  autofocus: true,
  onKeyEvent: (node, event) {
    if (event is! KeyDownEvent) return KeyEventResult.ignored;
    final isCtrl = PlatformUtils.isMacOS
        ? HardwareKeyboard.instance.isMetaPressed
        : HardwareKeyboard.instance.isControlPressed;
    
    // Ctrl+P: 命令面板
    if (event.logicalKey == LogicalKeyboardKey.keyP && isCtrl) {
      CommandPalette.show(context);
      return KeyEventResult.handled;
    }
    // Ctrl+F: 查找
    if (event.logicalKey == LogicalKeyboardKey.keyF && isCtrl) {
      ref.read(editorProvider.notifier).toggleFindReplace();
      return KeyEventResult.handled;
    }
    // Ctrl+H: 替换
    if (event.logicalKey == LogicalKeyboardKey.keyH && isCtrl) {
      ref.read(editorProvider.notifier).toggleFindReplace();
      return KeyEventResult.handled;
    }
    return KeyEventResult.ignored;
  },
  child: ...,
)
```

### 2.3 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 快捷键处理位置 | 顶层 `Focus` | 确保无论焦点在何处都能响应 |
| 平台适配 | `isMetaPressed`(macOS) vs `isControlPressed`(Win/Linux) | 符合各平台用户习惯 |
| 事件过滤 | 仅处理 `KeyDownEvent` | 避免 KeyUp/KeyRepeat 重复触发 |

## 3. 主题系统重构

### 3.1 问题分析

#### 原架构（V1.0.1）

```
AppConfig
├── themeName: String      // 用户选择的主题名称
└── darkMode: bool         // 独立的深色模式开关

app.dart
├── theme: AppTheme.getTheme(config.themeName)      // 浅色主题数据
├── darkTheme: AppTheme.getTheme('darkGraphite')     // 硬编码深色主题
└── themeMode: config.darkMode ? dark : light        // 根据开关切换
```

**问题**: `darkMode = true` + `themeName = 'shibuya'` 时：
- Flutter 使用 `darkTheme`（darkGraphite 的 ThemeData）
- 但 `AppTheme.getTokens('shibuya')` 返回浅色 tokens
- 组件中直接使用 tokens 的颜色与 ThemeData 冲突

#### 不可调和的矛盾
`darkMode` 开关与主题选择存在语义冲突：
- 主题本身已包含 brightness 信息（浅色/深色）
- 独立的 darkMode 开关创造了无效状态组合（浅色主题 + 深色模式）

### 3.2 新架构

```
AppConfig
└── themeName: String      // 唯一的主题选择，brightness 由主题决定

app.dart
├── theme: AppTheme.getTheme(config.themeName)       // 唯一主题
├── themeMode: tokens.brightness == dark ? dark : light  // 自动推断
└── (无 darkTheme)

AppTheme
├── lightThemeNames: ['redGraphite', 'shibuya']
└── darkThemeNames: ['darkGraphite', 'dieciOLED', 'nord']
```

### 3.3 数据模型变更

#### AppConfig (移除 darkMode)

```dart
// 移除的字段
// bool darkMode;

// 简化后的 fromJson
factory AppConfig.fromJson(Map<String, dynamic> json) {
  return AppConfig(
    themeName: json['themeName'] as String? ?? 'redGraphite',
    // darkMode 字段被完全移除
    ...
  );
}
```

#### SettingsProvider (移除 toggleDarkMode)

```dart
// 移除的方法
// Future<void> toggleDarkMode() async { ... }

// 简化后的 setTheme
Future<void> setTheme(String theme) async {
  await updateConfig((config) => config.copyWith(themeName: theme));
}
```

### 3.4 设置页面 UI 重新设计

```
主题
  ┌─ 浅色主题 ──────────────────────┐
  │  ┌──────────┐  ┌──────────┐    │
  │  │ Red      │  │ Shibuya  │    │
  │  │ Graphite │  │          │    │
  │  │  (默认)  │  │          │    │
  │  └──────────┘  └──────────┘    │
  └────────────────────────────────┘
  ┌─ 深色主题 ──────────────────────┐
  │  ┌──────────┐  ┌──────┐  ┌───┐ │
  │  │ Dark     │  │Dieci │  │Nord│ │
  │  │ Graphite │  │ OLED │  │   │ │
  │  └──────────┘  └──────┘  └───┘ │
  └────────────────────────────────┘
```

每个主题卡片包含：
- 主题预览色块（背景色 + 强调色）
- 主题名称（国际化）
- 选中状态（蓝色边框 + 勾选图标）

### 3.5 向后兼容

从 V1.0.1 升级到 V1.1.1 时：
- `AppConfig.fromJson()` 忽略 `darkMode` 字段（自动丢弃）
- 用户之前选择的 `themeName` 保持不变
- 如果之前是深色模式 + 浅色主题，升级后会显示浅色主题（符合主题本身的 brightness）

## 4. UI 高度优化

### 4.1 问题分析

| 组件 | Material 3 默认高度 | 目标高度 |
|------|-------------------|---------|
| MenuItemButton | ~48px | 36px |
| PopupMenuItem | ~48px | 36px |
| EditorTabBar | 44px | 44px（保持） |

### 4.2 全局主题方案

#### menuButtonTheme

```dart
menuButtonTheme: MenuButtonThemeData(
  style: ButtonStyle(
    minimumSize: WidgetStatePropertyAll(Size(0, 36)),
    padding: WidgetStatePropertyAll(
      EdgeInsets.symmetric(horizontal: 12, vertical: 8),
    ),
    textStyle: WidgetStatePropertyAll(
      TextStyle(fontSize: 14, fontWeight: FontWeight.normal, color: tokens.colorText),
    ),
    backgroundColor: WidgetStateProperty.resolveWith((states) {
      if (states.contains(WidgetState.hovered)) {
        return tokens.colorSurfaceHover;
      }
      return Colors.transparent;
    }),
    shape: WidgetStatePropertyAll(
      RoundedRectangleBorder(borderRadius: BorderRadius.circular(8)),
    ),
  ),
),
```

#### popupMenuTheme

```dart
popupMenuTheme: PopupMenuThemeData(
  textStyle: TextStyle(fontSize: 14, fontWeight: FontWeight.normal, color: tokens.colorText),
  shape: RoundedRectangleBorder(borderRadius: BorderRadius.circular(8)),
),
```

### 4.3 逐项设置方案

对于 `PopupMenuItem` 没有全局 height 设置机制，需在每个实例上单独设置：

```dart
PopupMenuItem(
  value: 'close',
  height: 36,
  child: Text(l10n.closeFile, style: const TextStyle(fontWeight: FontWeight.normal)),
),
```

### 4.4 设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| 目标高度 | 36px | 在紧凑性和可点击性之间取得平衡 |
| 菜单按钮高度方案 | 全局 ThemeData | 避免在每个 MenuItemButton 上重复设置 |
| 右键菜单高度方案 | 逐项设置 `height: 36` | PopupMenuItem 无全局 height 机制 |
| 标签页高度 | 保持 44px | 已经合理，且包含关闭按钮需要足够空间 |

## 6. 预览模式表格溢出修复

### 6.1 问题分析

**现象**：预览模式下，表格内容过长时溢出容器边界，导致布局错乱。

**根因**：`_buildTable()` 使用 `IntrinsicColumnWidth()` 作为列宽策略，当表格内容超出容器宽度时没有滚动机制。

### 6.2 解决方案

**文件**：`lib/ui/editor/markdown_renderer.dart`

**修改**：将 `Table` 包裹在 `SingleChildScrollView` 中

```dart
Widget _buildTable(md.TableNode node, ThemeData theme) {
  final colCount = node.headers.length;
  return Container(
    margin: const EdgeInsets.symmetric(vertical: 8),
    decoration: BoxDecoration(
      border: Border.all(color: theme.dividerColor),
      borderRadius: BorderRadius.circular(4),
    ),
    child: SingleChildScrollView(
      scrollDirection: Axis.horizontal,
      child: ConstrainedBox(
        constraints: const BoxConstraints(minWidth: 200),
        child: Table(
          border: TableBorder.symmetric(
            inside: BorderSide(color: theme.dividerColor),
          ),
          defaultColumnWidth: const IntrinsicColumnWidth(),
          children: [
            // ... 表格内容
          ],
        ),
      ),
    ),
  );
}
```

### 6.3 验证
- 长内容表格可水平滚动
- 短内容表格正常显示
- 表格保持圆角边框和分割线

## 7. 预览内容宽度自适应

### 7.1 问题分析

**现象**：隐藏侧边栏后，预览区域正文宽度不变，仅增大两侧边距。

**根因**：`markdown_renderer.dart` 的 `ConstrainedBox` 使用硬编码 `maxWidth: 680`，未使用 `AppConfig.editorMaxWidth` 配置值。

### 7.2 解决方案

**文件**：`lib/ui/editor/markdown_renderer.dart`

**修改前**：
```dart
constraints: const BoxConstraints(maxWidth: 680),
```

**修改后**：
```dart
final config = ref.watch(settingsProvider);
// ...
constraints: BoxConstraints(maxWidth: config.editorMaxWidth.toDouble()),
```

### 7.3 验证
- 预览内容宽度使用 `editorMaxWidth` 配置值（默认 800）
- 隐藏侧边栏后内容区域宽度增加
- 用户可在设置中调整最大宽度

## 8. 预览模式字体优化

### 8.1 问题分析

**现象**：预览模式正文使用 Flutter 默认系统字体，阅读体验差。

**根因**：`_buildParagraph()`、`_buildHeading()`、`_buildBlockquote()` 等方法的 `TextStyle` 未指定 `fontFamily`。

### 8.2 原版 MarkText 字体配置

参考 https://github.com/marktext/marktext 源码：

- **正文字体**：`"Open Sans", "Clear Sans", "Helvetica Neue", Helvetica, Arial, sans-serif`
- **代码字体**：`"DejaVu Sans Mono", "Source Code Pro", "Droid Sans Mono", Consolas, monospace`
- **字号**：16px
- **行高**：1.6

### 8.3 解决方案

**文件**：`lib/ui/editor/markdown_renderer.dart`

**添加常量**：
```dart
static const _previewFontFamily = 'Open Sans, Helvetica Neue, Arial, sans-serif';
```

**修改点**：

| 方法 | 修改内容 |
|------|----------|
| `_buildParagraph()` | `fontFamily: _previewFontFamily, fontSize: 16, height: 1.6` |
| `_buildHeading()` | 所有 4 个级别添加 `fontFamily: _previewFontFamily` |
| `_buildBlockquote()` | `fontFamily: _previewFontFamily, fontSize: 16, height: 1.6` |
| `_buildTable()` | 表头和表体文本添加 `fontFamily: _previewFontFamily` |

### 8.4 验证
- 预览模式使用 Open Sans 字体族
- 段落字号 16px，行高 1.6
- 标题、引用块、表格文本字体一致

## 10. 技术债务

### 10.1 已知限制
1. **窗口标题栏颜色**: 系统级窗口标题栏颜色无法跟随主题变化（需要 `window_manager` 包，但已在 V1.0.1 BUG-001 中移除）
2. **PopupMenuItem 全局高度**: Flutter 不支持通过 ThemeData 全局设置 PopupMenuItem 高度，需逐项设置
3. **字体回退**: Open Sans 字体需要系统安装，未安装时会回退到 Helvetica Neue 或 Arial

### 10.2 未来改进方向
1. 探索使用 `flutter_window_close` 或类似轻量包来控制窗口标题栏颜色
2. 封装自定义 PopupMenuItem wrapper 统一高度设置
3. 考虑将 Open Sans 字体打包到应用中，确保跨平台一致性
